# Technology Stack — logpose v0.2.0 (incremental additions)

**Project:** logpose (rename/generalize of `palworld-server-launcher` v0.1.19)
**Researched:** 2026-04-12
**Scope:** Incremental stack bits needed for ARK: Survival Evolved support + multi-game Typer dispatch. Existing stack (Python 3.8+, Typer, Rich, subprocess, systemd, Polkit, SteamCMD) stays.

**Overall confidence:** HIGH for ARK server specifics and Typer pattern (verified against official docs/wiki). MEDIUM for one detail corrected against PROJECT.md (RCON default port).

## TL;DR — What to add, what not to add

| Need | Recommendation | Add new dep? |
|------|----------------|--------------|
| ARK `GameUserSettings.ini` parse/write | Stdlib `configparser.ConfigParser(strict=False, interpolation=None, comment_prefixes=(';',))` | **No** |
| ARK server install | Existing `steamcmd` path + `app_update 376030`; add `lib32gcc-s1` + `libc6-i386` to apt step | **No** (apt packages only) |
| Multi-game CLI dispatch | `typer.Typer()` per game + `app.add_typer(game_app, name="palworld")` in root | **No** |
| PyPI distribution name | `logpose-launcher` (the name `logpose` is taken on PyPI) | — |
| Python version floor | Keep `>=3.8` **and pin** `typer>=0.9,<0.21` + `rich<14` — OR — bump to `>=3.10` and use current Typer | Pin only |

**Do NOT add** `configobj`, `iniparse`, `configupdater`, `python-a2s`, `gamedig`, or any RCON library. None are needed for v0.2.0 scope (no server-query or RCON features shipped). Stdlib `configparser` covers ARK's INI format cleanly for the edit-settings use case.

## 1. ARK `GameUserSettings.ini` — stdlib `configparser` recommendation

**Decision:** Use `configparser.ConfigParser` from the stdlib. **Confidence: HIGH.**

### Instantiation (exact form for ARK)

```python
import configparser

parser = configparser.ConfigParser(
    strict=False,          # tolerate duplicate keys (defensive; default ARK files have none)
    interpolation=None,    # ARK values may contain '%' in message-of-the-day text
    comment_prefixes=(';',), # ARK uses ';' inline; leave '#' out so it isn't consumed
    allow_no_value=True,   # some ARK keys appear bare in community configs
    delimiters=('=',),     # only '=' — don't let ':' get treated as a delimiter
)
parser.optionxform = str   # preserve case; ARK keys are CamelCase
parser.read(gameusersettings_path, encoding='utf-8')
```

### ARK INI dialect observations (verified)

| Concern | Reality | Source |
|--------|---------|--------|
| Section headers present | Yes — `[ServerSettings]`, `[SessionSettings]`, `[/Script/ShooterGame.ShooterGameUserSettings]`, `[/Script/Engine.GameSession]`, `[MessageOfTheDay]`, `[Ragnarok]`, etc. | [ARK Wiki: Server configuration](https://ark.wiki.gg/wiki/Server_configuration) |
| Duplicate keys in default file | No (in default). Community configs occasionally have them (e.g. repeated `ConfigOverrideItemMaxQuantity`). `strict=False` is defensive. | [CPython #107428](https://github.com/python/cpython/issues/107428), [Akliz docs](https://help.akliz.net/docs/find-gameusersettingsini-on-an-ark-survival-evolved-server) |
| `%` characters in values | Possible in `[MessageOfTheDay] Message=` (e.g. `"50% off tames"`). Must disable interpolation. | Python docs — `BasicInterpolation` escapes `%` as `%%` |
| Unquoted values with `=` | Not observed; ARK uses single `=` as key/value separator. `configparser` uses first `=` as delimiter so even if a value contained `=` it would be handled. | [Python configparser docs](https://docs.python.org/3/library/configparser.html) |
| Spaces in values | Yes in `SessionName`, `Message`, map overrides. `configparser` strips surrounding whitespace but preserves internal spaces — OK. | Python docs |
| Case sensitivity | ARK keys are CamelCase (`DifficultyOffset`, `RCONPort`). Default `configparser` lowercases keys via `optionxform` — **override to `str`** to preserve. | Python docs |
| Comment preservation on write | **Not preserved.** `configparser` drops comments on round-trip. This is the one real limitation. | Python docs (explicit) |

### Comment-preservation limitation — acceptable for v0.2.0

**Trade-off:** The existing Palworld `edit-settings` flow uses regex substitution over a single `OptionSettings=(...)` line, which preserves surrounding comments. The ARK adapter using `configparser` will **drop comments on write**.

**Why this is acceptable:**
- ARK's default `GameUserSettings.ini` generated by the server on first run contains zero comments (verified via Steam community default dump).
- Users who hand-edit and add comments are a rare sub-case; for v0.2.0 scope (standard installs, scripted edit flow) this is acceptable.
- Third-party libraries (`configupdater`, `configobj`) would add a runtime dep and violate the "no new deps unless strongly justified" constraint.

**If comment preservation ever becomes required** (future milestone): swap in `configupdater` (PyPA-adjacent, BSD licensed, ~200 KB, pure-Python). Not now.

### What NOT to use

| Library | Why reject |
|---------|-----------|
| `configobj` | Unmaintained-ish (last release 2023), extra dep for marginal benefit over stdlib |
| `iniparse` | Obsolete, last release ~2014 |
| A custom line-regex parser (like the Palworld one) | Unnecessary — ARK's INI **is** standard INI. Don't hand-roll what stdlib handles. |

### Rebuild/regenerate fallback (mirror Palworld's `_create_settings_from_default`)

ARK regenerates `GameUserSettings.ini` on first server launch if missing. No need for a "seed from default template" equivalent — just start the server once (or tell the user to). The adapter only needs to work **after** the server has run once.

## 2. ARK: Survival Evolved on Linux via SteamCMD

### Verified facts

| Item | Value | Confidence | Source |
|------|-------|-----------|--------|
| SteamCMD app id | `376030` | HIGH | [SteamDB app 376030](https://steamdb.info/app/376030/info/), [ARK Wiki](https://ark.wiki.gg/wiki/Dedicated_server_setup) |
| Binary path (relative to install dir) | `ShooterGame/Binaries/Linux/ShooterGameServer` | HIGH | ARK Wiki, LinuxGSM, Akliz, ServerMania |
| Default GamePort | `7777/udp` | HIGH | ARK Wiki |
| Default QueryPort | `27015/udp` | HIGH | ARK Wiki |
| **Default RCONPort** | **`27020/tcp`** | HIGH | Python-verified: [ARK Wiki Server configuration](https://ark.wiki.gg/wiki/Server_configuration), [Steam default GameUserSettings dump](https://steamcommunity.com/app/346110/discussions/0/535151589892828308/) |
| Default RCONEnabled | `false` | HIGH | Same |
| Linux glibc requirement | `>=2.17` | HIGH | ARK Wiki |

### ⚠️ CORRECTION to PROJECT.md

PROJECT.md (line 35 of the Active requirements) says "RCON 32330 optional" as the default. **This is wrong.** The ARK default is **`27020`**, not `32330`. 32330 is a *popular user-chosen override* (seen in many third-party hosting guides because they want to avoid port collisions with Steam's query range), not the server's default. Recommend the roadmap use `27020` as the template default and let operators override via the CLI arg.

### Launch arguments — verified format

Ordering: **map + `?` options first, then `-` flags separately, terminated by `-server -log`.**

```
./ShooterGameServer TheIsland?listen?SessionName={session_name}?ServerPassword={server_password}?ServerAdminPassword={admin_password}?Port={port}?QueryPort={query_port}?MaxPlayers={max_players} -server -log
```

**Notes:**
- `listen` is mandatory (makes it a listen server, not a client).
- `?` options are ARK-specific config overrides; `-` flags are Unreal Engine runtime flags.
- `MaxPlayers` (not `WinLiveMaxPlayers`) is correct for ASE — verified on ark.wiki.gg.
- If RCON is enabled via `?RCONEnabled=True?RCONPort=27020`, append those to the `?`-chain (or set them in `GameUserSettings.ini`'s `[ServerSettings]` section — both work; CLI takes precedence).
- **ASE (app 376030)** uses `?Port=`. **ASA (app 2430930)** uses `-port=`. Since v0.2.0 is ASE-only per PROJECT.md, this is not a concern.
- The `-UseBattlEye` / `-NoBattlEye` flag is optional; ARK has BattlEye on by default. If clients can't connect, recommend `-NoBattlEye`.

### Extra apt packages required for ARK (on top of `steamcmd`)

The `steamcmd` package on Debian/Ubuntu already pulls in most 32-bit deps, but ARK's `ShooterGameServer` binary itself is 64-bit and needs additional libraries that `steamcmd`'s dep chain doesn't guarantee:

| Package | Why | Confidence |
|---------|-----|-----------|
| `lib32gcc-s1` | 32-bit GCC support lib needed by Steam client libraries loaded by the server. **Ubuntu 22.04+/Debian 12+** name. | HIGH |
| `lib32stdc++6` | 32-bit C++ runtime (same reason). | HIGH |
| `libc6-i386` | 32-bit glibc. | HIGH |
| `libsdl2-2.0-0` | Server uses SDL for input device enumeration even headless — missing it causes cryptic startup failures. | MEDIUM |
| `libssl-dev` (or `libssl3`) | Occasionally reported missing on minimal cloud images. | LOW (fallback) |
| `curl`, `wget`, `tar`, `unzip` | Already pulled by `steamcmd` but list defensively. | — |

**Older systems:** On Ubuntu 20.04 and earlier `lib32gcc-s1` doesn't exist — the package is `lib32gcc1`. Since PROJECT.md targets Debian/Ubuntu **generally** and GCP/cloud Debian images are often 11/12 or Ubuntu 22.04/24.04, the recommendation is:

```bash
# Add i386 architecture first (required)
sudo dpkg --add-architecture i386
sudo apt-get update
# Then install (with lib32gcc-s1 fallback to lib32gcc1 if not found)
sudo apt-get install -y lib32gcc-s1 lib32stdc++6 libc6-i386 libsdl2-2.0-0 || \
sudo apt-get install -y lib32gcc1 lib32stdc++6 libc6-i386 libsdl2-2.0-0
```

The `_install_steamcmd()` path in `main.py` already calls `sudo dpkg --add-architecture i386`. Add a new `_install_ark_prereqs()` helper (or extend `_install_steamcmd()` conditionally) to apt-install the ARK-specific libs.

**Sources:** [ARK Wiki dedicated server setup](https://ark.wiki.gg/wiki/Dedicated_server_setup), [arkmanager/ark-server-tools README](https://github.com/arkmanager/ark-server-tools), [Steam Community installation guide](https://steamcommunity.com/sharedfiles/filedetails/?id=828200218)

### Install invocation (mirror existing Palworld pattern)

```bash
steamcmd +force_install_dir '/home/{user}/.steam/steam/steamapps/common/ARKServer' \
         +login anonymous \
         +app_update 376030 validate \
         +quit
```

After install: `chmod +x ShooterGame/Binaries/Linux/ShooterGameServer`.

### `steamclient.so` fix (ARK analog of existing `_fix_steam_sdk()`)

ARK has the same Steamworks SDK redist issue as Palworld. The existing `_fix_steam_sdk()` helper copies `Steamworks SDK Redist/linux64/steamclient.so` to `~/.steam/sdk64/`. **This is sufficient for ARK too** — the SDK redist is a shared SteamCMD component, not per-game. No new code needed; call the existing helper from the ARK install path.

### systemd unit — `arkserver.service` template (new file needed)

Mirror `palserver.service.template`:

```ini
[Unit]
Description=ARK: Survival Evolved Dedicated Server
After=network.target

[Service]
User={user}
Group={user}
Type=simple
WorkingDirectory={working_directory}
ExecStart={exec_start_path} {map}?listen?SessionName={session_name}?ServerPassword={server_password}?ServerAdminPassword={admin_password}?Port={port}?QueryPort={query_port}?MaxPlayers={max_players} -server -log
Restart=always
RestartSec=180
LimitNOFILE=100000

[Install]
WantedBy=multi-user.target
```

`LimitNOFILE=100000` is an ARK-community-recommended addition (ARK keeps many file handles open for save/biome caches). Palworld doesn't need it; add it only on the ARK unit.

### Polkit rule — merged or per-game?

Current `palserver.rules.template` hardcodes `action.lookup("unit") == "palserver.service"`. For two games, the cleanest minimum-diff approach is to **keep the existing `40-palserver.rules` file unchanged** and **add a new `40-arkserver.rules`** with the same template but `unit == "arkserver.service"`. This preserves behavioral compatibility (PROJECT.md constraint) and avoids touching a working file.

**Alternative** (if roadmap prefers one file): expand the template to accept a list of service names. Palworld-only installs then get `["palserver.service"]`, ARK-only get `["arkserver.service"]`, both-installed get both. This is a nice-to-have; per-file is simpler and the minimum diff.

## 3. Typer nested subcommands — canonical pattern

**Confidence: HIGH.** Verified against [Typer tutorial source on GitHub](https://github.com/fastapi/typer/tree/master/docs_src/subcommands).

### Canonical pattern

The idiomatic way is **one `typer.Typer()` instance per game namespace**, added to the root app via `app.add_typer(sub_app, name="...")`.

```python
# logpose/main.py
import typer

from logpose.games import palworld, ark  # sibling modules each exposing .app

app = typer.Typer(help="logpose — multi-game dedicated server launcher.")
app.add_typer(palworld.app, name="palworld", help="Manage a Palworld dedicated server.")
app.add_typer(ark.app,      name="ark",      help="Manage an ARK: Survival Evolved dedicated server.")

if __name__ == "__main__":
    app()
```

```python
# logpose/games/palworld.py
import typer
app = typer.Typer()

@app.command()
def install(port: int = 8211, players: int = 32, start: bool = False) -> None:
    """Install the Palworld dedicated server."""
    ...

@app.command()
def start() -> None:
    """Start the Palworld server."""
    ...

# (and so on for stop, restart, status, enable, disable, update, edit_settings)
```

```python
# logpose/games/ark.py
import typer
app = typer.Typer()

@app.command()
def install(
    map: str = typer.Option("TheIsland", help="ARK map name."),
    session_name: str = typer.Option("logpose ARK Server", help="Public session name."),
    port: int = typer.Option(7777),
    query_port: int = typer.Option(27015),
    rcon_port: int = typer.Option(27020),
    max_players: int = typer.Option(70),
    server_password: str = typer.Option("", help="Join password (blank = open)."),
    admin_password: str = typer.Option(..., prompt=True, hide_input=True),
    start: bool = typer.Option(False, "--start"),
) -> None:
    """Install the ARK: Survival Evolved dedicated server."""
    ...
```

Invocation: `logpose palworld install --port 8211`, `logpose ark install --map Ragnarok`.

### Gotchas

1. **Help output shows sub-apps as command groups** — confirmed from Typer docs. Each `add_typer` creates a "Commands" section row. If you want a description for the group on the root help, pass `help=...` to `add_typer()` (shown above), not just to `Typer()` inside the submodule. Both together is fine.
2. **Shared options don't auto-propagate** — if both games need the same option (rare in v0.2.0), you either declare it on each command or use a **callback** (`@app.callback()`) on a sub-app that takes the option and stuffs it into `ctx.obj`. For v0.2.0 each game has different install params, so skip callbacks.
3. **Name collisions** — `name="ark"` must be distinct from any top-level `@app.command()` on the root. Since the current root has flat commands (`install`, `start`, etc.) that you're moving into sub-apps, you'll be **removing** those decorators from the root; the root should only contain `add_typer` calls and nothing else (or a single `@app.callback()` for a `--version` flag). Important: if the root has zero commands after the refactor, Typer will still work — nested sub-apps are sufficient.
4. **`invoke_without_command=True`** is unnecessary; the default already makes `logpose` (no args) print help.
5. **Typer version** — `add_typer()` has been stable since Typer 0.0.x; no version-specific concerns within the supported band.

### File structure recommendation (respecting PROJECT.md's "no `core/` split")

PROJECT.md explicitly rejects a `core/` module split. To honor that **and** get the Typer nested-subcommand ergonomics, do the simplest thing:

```
logpose/
├── __init__.py
├── main.py            # root Typer app + add_typer calls + GAMES dict + all helpers
├── games/
│   ├── __init__.py
│   ├── palworld.py    # Typer sub-app, imports helpers from main
│   └── ark.py         # Typer sub-app, imports helpers from main
└── templates/
    ├── palserver.service.template
    ├── palserver.rules.template
    ├── arkserver.service.template
    └── arkserver.rules.template
```

Or even simpler — **all in `main.py`** (two sub-app `typer.Typer()` instances declared at module scope, commands decorated on each). ~500 lines total, still one file. This is closer to the current "single-file implementation" and matches the minimum-diff constraint best. Pick this unless `main.py` exceeds ~800 lines post-refactor.

## 4. PyPI publishing — `logpose` distribution name

**Confidence: HIGH** for the availability check procedure and the name result.

### Canonical availability check (authoritative)

PyPI has no first-party "is this name available" API, but the **Simple Index** is the authoritative source pip uses:

```bash
# Authoritative check — returns HTTP 200 if taken, 404 if available
curl -sI https://pypi.org/simple/<name>/ | head -n1
```

Or the JSON metadata endpoint:

```bash
curl -sI https://pypi.org/pypi/<name>/json | head -n1
```

Both return `HTTP/2 200` for existing names and `HTTP/2 404` for free names. This is how `twine upload` effectively determines collisions at publish time (server-side 403 with "File already exists" for the exact version, or 400 "The user isn't allowed to upload to project" for an existing project owned by someone else).

### Result of check (run 2026-04-12)

| Name | Status | Notes |
|------|--------|-------|
| `logpose` | **TAKEN** — v0.0.1b2 by Giuseppe Savino, "A python log library for data science" | pypi.org/project/logpose/ returns 200 |
| `logpose-launcher` | **AVAILABLE** | pypi.org/simple/logpose-launcher/ returns 404 |
| `logpose-server-launcher` | **AVAILABLE** | pypi.org/simple/logpose-server-launcher/ returns 404 |

**Recommendation: use `logpose-launcher`** as the PyPI distribution name.

### Pyproject.toml update (distribution name vs import name)

The **distribution name** (pip install target) and **import name** (Python module) can differ. Recommended:

```toml
[project]
name = "logpose-launcher"           # what users `pip install`
version = "0.2.0"
# ...

[project.scripts]
logpose = "logpose.main:app"         # CLI command stays short

[tool.setuptools]
packages = ["logpose"]               # import name stays `logpose`
```

Users do `pip install logpose-launcher`, then run `logpose palworld install`. The CLI entry point name (`logpose`) is independent of the distribution name — this is the same trick `typer-cli` / `typer` and many others use.

### Python 3.8 compatibility — a real concern

**The existing `pyproject.toml` carries `requires-python = ">=3.8"` with unpinned `typer` / `rich`.** This is a latent bug today:

| Package | Current latest | `requires-python` | Last version supporting 3.8 |
|---------|----------------|-------------------|-----------------------------|
| `typer` | 0.24.1 | `>=3.10` | **0.20.1** (released 2025-12-19; 0.21.0 dropped 3.8 on 2025-12-25) |
| `rich`  | 15.0.0 | `>=3.9`  | last 3.8-compat ~`rich 13.8.x` |

A fresh `pip install palworld-server-launcher` on a Python 3.8 box today will fail dependency resolution or pull old versions. For v0.2.0, **pick one**:

**Option A (minimum diff, preserve 3.8):** Pin dependencies in `pyproject.toml`:
```toml
requires-python = ">=3.8"
dependencies = [
    "typer>=0.9,<0.21",
    "rich>=13.0,<14",
]
```
Keeps 3.8 users on GCP's default Python working. Matches existing `>=3.8`.

**Option B (bump floor):** Move to `requires-python = ">=3.10"` and use current Typer/Rich. Cleaner long-term, but breaks any existing 3.8 user.

**Recommendation: Option A.** Debian 11 (bullseye, still supported until 2026) ships Python 3.9 by default, but plenty of cloud images pin older. The existing tool claimed 3.8+ so honor it. No code change required in `main.py` — only `pyproject.toml` dependency version pins.

### Python 3.8 code-level caveats (verify during execution phase)

- `main.py` uses `dict[str, str]` PEP-585 generics (line 189, 202, 253, 267). **Requires `from __future__ import annotations`** on Python 3.8 or these become runtime `TypeError`s. The existing v0.1.19 ships without that import — either it's untested on 3.8 or runs OK because the annotations are on function signatures that Typer inspects lazily. Either way, **add `from __future__ import annotations`** to every module during the refactor. Zero cost, future-proofs the generics.
- Walrus `:=` is fine (3.8+).
- No `match` statements — good, those are 3.10+.
- `Optional[int] = None` style still works universally.

## Alternatives considered and rejected

| Category | Recommended | Rejected alternative | Why rejected |
|----------|-------------|----------------------|--------------|
| INI parser | `configparser` (stdlib) | `configobj` | Adds dep; ARK's INI is standard; `configobj`'s comment preservation not needed in v0.2.0 |
| INI parser | `configparser` | `configupdater` | Adds dep; only buys comment preservation; defer until users demand it |
| CLI nesting | `add_typer()` per game | `--game` flag on flat commands | Less ergonomic; PROJECT.md explicitly chose game-first nesting (kubectl-style) |
| CLI nesting | `add_typer()` per game | Click-native groups | Already on Typer; Typer wraps Click cleanly for this pattern |
| Dist name | `logpose-launcher` | `logpose` | Taken on PyPI by an unrelated log library |
| Dist name | `logpose-launcher` | `logpose-server-launcher` | Longer, redundant; both are available but shorter wins |
| RCON | None in v0.2.0 | `rcon`, `aiomcrcon`, `mcrcon` | Not in scope; GameUserSettings.ini edit covers v0.2.0 config needs |
| Python min | 3.8 with pinned deps | Bump to 3.10 | Would break existing implied support; minimum diff wins |
