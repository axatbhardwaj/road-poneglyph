<objective>
Publish `logpose-launcher` 0.2.0 to production PyPI, completing the v0.2.0 milestone. After this plan, anyone can `pip install logpose-launcher` on a fresh machine and get the multi-game CLI.

Purpose: closes PKG-07 (published to PyPI as logpose-launcher v0.2.0; old palworld-server-launcher v0.1.19 untouched on PyPI).

This plan is HUMAN-GATED: it requires a production PyPI API token. Unlike TestPyPI, PyPI uploads are IRREVERSIBLE — a 0.2.0 upload with a bug stays published forever (yanking hides but does not delete; the filename is permanently claimed). That is why Plan 06-03 (TestPyPI dry-run) is a hard dependency.

Output: `logpose-launcher` 0.2.0 live on PyPI; `pip install logpose-launcher` works globally.
</objective>

  <how-to-verify>
    Run on the machine that has production PyPI access:

    1. Get a production PyPI API token (distinct from the TestPyPI token in Plan 06-03):
       - Sign in at https://pypi.org/ (this is a DIFFERENT account system from TestPyPI — the login you already have on one is not valid on the other).
       - https://pypi.org/manage/account/token/ → "Add API token".

    2. From the repo root, upload both artefacts:
       ```bash
       export TWINE_USERNAME=__token__
       export TWINE_PASSWORD='<paste-the-production-PyPI-token>'

       unset TWINE_PASSWORD
       ```

       Note: NO `--repository` flag — the twine default is production PyPI. Passing `--repository testpypi` here would push to TestPyPI instead.

    3. Expected twine output (final two lines):
       ```
       View at:
       https://pypi.org/project/logpose-launcher/0.2.0/
       ```

    4. Open https://pypi.org/project/logpose-launcher/0.2.0/ in a browser. Confirm:
       - Project page loads (HTTP 200).
       - Title shows `logpose-launcher 0.2.0`.
       - README renders as Markdown with the multi-game content (same checks as Plan 06-03 Task 2, now against production).
       - "Download files" section lists both the `.whl` and `.tar.gz`.
       - File hashes on the PyPI page match the local `sha256sum` output from Task 1.

    5. Open https://pypi.org/project/palworld-server-launcher/ — verify the "latest release" still shows 0.1.19 and the release date is unchanged from the original v0.1.19 publish.

    6. If the upload fails:
       - "File already exists" on a first 0.2.0 push: somebody (possibly you, on another machine) already uploaded 0.2.0. Do NOT rebuild — inspect https://pypi.org/project/logpose-launcher/0.2.0/ first. If it is the artefact we verified, skip to Task 3. If not, escalate (hash mismatch means we have a provenance problem, not a retry candidate).
       - Auth failure (403): confirm the token is a production PyPI token (starts with `pypi-` from pypi.org, not test.pypi.org), and that the token scope permits uploading to `logpose-launcher` (or is "Entire account").
       - Any other error: paste the full twine output into your resume-signal message so we can diagnose.

    7. Unset `TWINE_PASSWORD` after upload. Do NOT add the token to any shell history file or dotenv.
