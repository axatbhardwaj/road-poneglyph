<objective>
Dry-run publish `logpose-launcher` 0.2.0 to TestPyPI and prove that a fresh user, on a fresh machine, can `pip install` it from the TestPyPI index and land a working `logpose` CLI. This is the dress rehearsal before production PyPI (Plan 06-04).

Purpose: closes E2E-06 (TestPyPI dry-run publish + throwaway-venv install). Catches metadata / package-data / entry-point bugs BEFORE they hit the real index, where they'd be frozen forever (PyPI filenames are immutable).

Output: `logpose-launcher` 0.2.0 visible on https://test.pypi.org/project/logpose-launcher/, README rendered correctly, `logpose --help` inside a throwaway venv shows both sub-commands.

This plan is HUMAN-GATED: it requires a TestPyPI API token (`TWINE_PASSWORD`). The checkpoint structure splits the plan into (a) autonomous pre-flight that Claude runs up to the upload line, (b) the human-action step for the token, and (c) autonomous post-upload verification.
</objective>

<!-- TestPyPI publishing surface. -->
<!-- Executor uses twine against the TestPyPI legacy endpoint, not pypi.org. -->

TestPyPI endpoints:
  - Upload: https://test.pypi.org/legacy/          (twine --repository testpypi)
  - Simple index (pip):  https://test.pypi.org/simple/
  - Project page: https://test.pypi.org/project/logpose-launcher/

Twine authentication:
  - Username: __token__  (literal)
  - Password: pypi-XXXXXXXX... (API token, 'pypi-' prefix)
  - Passed via TWINE_USERNAME / TWINE_PASSWORD env vars OR --username / --password flags
  - Never committed to git; never echoed into logs

Key sharp edges:
  - TestPyPI does NOT mirror PyPI — `typer` and `rich` are NOT on TestPyPI. When installing from TestPyPI, add `--extra-index-url https://pypi.org/simple/` so pip resolves runtime deps from production PyPI.
  - Version numbers are immutable on TestPyPI too. If the 0.2.0 upload has a bug, the next attempt must be 0.2.0.post1 or 0.2.1. Plan 06-04 (production) MUST wait for a clean 0.2.0 here; if we burn 0.2.0 on TestPyPI, production is unaffected (separate indices), but the TestPyPI verification for version 0.2.0 is gone for that project.
  - The throwaway venv MUST be outside the repository tree to avoid accidentally picking up the source install. Recommended: `mktemp -d` → `python -m venv ...` inside that directory.

  <action>

       ```bash

       python --version   # confirm Python 3.8+
       ```

    2. Install from TestPyPI, with production PyPI as the fallback for runtime deps (TestPyPI does not mirror typer/rich):
       ```bash
       pip install --index-url https://test.pypi.org/simple/ \
                   --extra-index-url https://pypi.org/simple/ \
                   "logpose-launcher==0.2.0"
       ```

       Expected tail of output: `Successfully installed logpose-launcher-0.2.0 typer-X.Y.Z rich-A.B.C ...`

    3. Smoke-test the CLI inside the venv:
       ```bash
       which logpose                # must point inside $VENV_DIR/bin
       logpose --version            # must print 0.2.0
       logpose --help               # must list both `palworld` and `ark` as sub-commands
       logpose palworld --help      # must list the 9 Palworld verbs
       logpose ark --help           # must list the 11 ARK verbs
       logpose ark install --help   # must show all 12 flags (map/port/query-port/rcon-port/players/session-name/admin-password/password/beta/enable-autostart/start/generate-password)
       ```

       Capture the stdout of each command — these are the E2E-06 evidence strings.

    4. Confirm the installed wheel's templates are present in the venv's site-packages:
       ```bash
       python -c "from importlib.resources import files; print(sorted(p.name for p in files('logpose').joinpath('templates').iterdir()))"
       ```
       Expected list (Python 3.9+ resources API — on 3.8 use `importlib_resources` or fall back to `pathlib`):
       `['40-logpose.rules.template', 'CLAUDE.md', 'arkserver.service.template', 'logpose-ark.sudoers.template', 'palserver.service.template']`
       (CLAUDE.md presence is a known package-data quirk — acceptable for v0.2.0 but note it in the SUMMARY as a deviation if still present.)

    5. Deactivate + clean up:
       ```bash
       deactivate
       rm -rf "$(dirname "$VENV_DIR")"
       ```

    6. Do NOT install logpose-launcher globally or into the dev environment. The throwaway venv is the only install surface for this verification.

    If ANY smoke-test command fails or produces unexpected output, record the full stderr in the plan's SUMMARY under "Deviations" and escalate. Do NOT proceed to Plan 06-04 until this plan is green.
